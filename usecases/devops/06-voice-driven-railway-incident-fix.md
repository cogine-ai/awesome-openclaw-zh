# 语音驱动 Railway 故障定位与修复

> 边走路边语音下达任务，让 OpenClaw 完成“排障→修复→验证→PR”。

## 这个案例能帮你做什么

- 将线上故障处理拆成标准链路，减少漏步骤。
- 在移动场景下也能完成完整修复流程。
- 让修复过程可追溯，便于复盘与交接。

## 你需要的 Skills（按类型）

| 类型 | Skill / 工具 | 用途 | 来源 |
|---|---|---|---|
| 外部 | `Railway CLI` | 读取服务构建状态并更新配置 | Railway |
| 外部 | `GitHub CLI Read PR` | 提交修复方案与代码变更 | GitHub |
| 内置 | `语音指令` | 移动场景下连续下达任务 | OpenClaw Built-in |

## 快速体验版（先跑一轮）

1. 选择一个非核心服务做演练。
2. 让 OpenClaw 按“日志→根因→修复→验证”输出行动列表。
3. 你只在关键动作前确认。

```text
请按故障处理顺序执行：
1) 检查部署状态
2) 汇总失败日志
3) 给出根因判断
4) 提交最小修复方案
5) 重新部署并验证
6) 输出复盘记录
```

## 稳定自动版（可长期运行）

### 推荐流程

1. 固化故障模板：现象、影响、日志、根因、修复、验证。
2. 把“配置修复”和“代码修复”分轨处理。
3. 所有修复结论进入同一复盘文档。

## 成功标准

- [ ] 每次故障都有结构化处理记录。
- [ ] 修复后有明确验证结果。
- [ ] 关键配置改动保留人工确认。
